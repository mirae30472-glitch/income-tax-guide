소득세법 AI 검색·결산 지원 — 구축 가이드 v2

회계 실무 효율화·법령 검색을 목표로 한 단계별·비개발자 친화 로드맵. 원본 PDF의 RAG 구상은 3단계로 두고, 1단계는 오늘부터 운영 가능한 형태로 정리했습니다.

문서 안내

이 HTML은 프로젝트 폴더에 둔 구축가이드.pdf(RAG 아키텍처·LangChain·벡터 DB·Streamlit 등)를 원천(소스)으로 옮겨, 코딩을 하지 않는 직원·바이브 코딩 환경에서 무엇을 먼저 할지 나눈 문서입니다. 내용을 고칠 때는 먼저 같은 폴더의 PDF와 대조합니다.

포함: 목차 클릭 이동, 상단 검색으로 목차 필터, 업데이트 경고, 하단 메모란(브라우저에 임시 저장).

업데이트·유의사항

반드시 확인: 세법·예규는 개정됩니다. 이 문서와 연결된 가이드북·AI 답변은 항상 시행일·귀속 연도를 확인하세요.
  • 문서 상단 또는 배너에 “기준일 / 귀속연도”를 적어 두는 습관을 만드세요.
  • 개정 반영 후에는 버전 번호(예: v2.1)와 변경 요약을 남기세요.
  • AI·RAG 답변은 출처 조문·원문과 대조하지 않으면 사용하지 마세요.
법적·실무 책임: 본 가이드·자동 생성 답변은 참고용입니다. 신고·컨설팅 최종 판단은 세무사·담당자의 업무입니다.

3단계 로드맵 (현실版)

1단계지금 할 수 있는 것 (필수·기본)

원본 PDF의 「데이터 정제·조문화」 중 사무실이 직접 통제 가능한 부분만 가져옵니다.

  • 조문·절 차원의 목차와 앵커 링크(제목 클릭 시 해당 절로 이동).
  • 표는 HTML 표 또는 Markdown으로 원문 구조 유지 (엑셀 캡처만 올리면 검색·접근성이 나빠짐).
  • 검색창: 목차·본문 키워드로 빠르게 찾기 (현재 소득세_가이드북/index.html 패턴).
  • 메모란: 실무자 코멘트·사례를 카드별로 남기기.
  • 출처 필드: 각 토픽에 「법령명·조문·확인일」을 텍스트로 고정 기재.

→ 구현 가능: 코딩 없이도 Cursor 등으로 HTML을 유지보수하는 방식이면 충분히 가능합니다.

2단계경량 AI (선택)

원본의 「시맨틱 검색·자연어 질문」에 가깝게 가려면, 풀 RAG 이전에 아래부터 검토합니다.

  • 사내 허용 범위에서 문서 업로드형 챗(상용 서비스)으로 PDF·내부 요약을 올려 Q&A.
  • 또는 Streamlit 등 간단 UI + 로컬 벡터 DB — 이 경우 Python 환경을 다룰 수 있는 인력이 필요합니다.
  • 공통 원칙: 답변마다 출처(조문·페이지·내부 문서명)를 붙이는 규칙을 강제합니다.

3단계풀 RAG·자동 개정 파이프라인 (전문)

원본 PDF의 LangChain/LlamaIndex, Pinecone/Chroma, 자동 업데이트 파이프라인·피드백 루프는 이 단계입니다.

  • 법령 버전 관리(시행일, 개정 전후) 설계 없이 자동화하면 오히려 리스크가 큽니다.
  • 예규·질의회신·판례 연동은 저작권·이용약관·수집 범위를 사전에 정리해야 합니다.
  • 비개발 조직은 외주·내부 개발자와 SOW(기능·출처 정확도·갱신 주기)로 맞추는 것을 권장합니다.

리스크·책임 한도 (악마의 대변인)

  • RAG는 “정답 장치”가 아님: 잘못된 청크 조합·구법 데이터가 섞이면 신고 오류로 이어질 수 있습니다.
  • 자동 개정 파이프라인: 시행일·조문 번호 변경을 놓치면 가장 위험합니다. 사람의 검수 게이트가 필요합니다.
  • 예규·판례 수집: 무단 전문 수집·재배포는 법적·약관 리스크가 있습니다. 공식 출처·라이선스를 확인하세요.

MCP·Cursor·스킬 (비개발자 안내)

필수는 아닙니다. 1단계 HTML만으로도 가이드 운영은 가능합니다.

  • Cursor: CLAUDE.md에 “HTML만 사용, 목차·검색·메모·업데이트 문구 유지”처럼 규칙을 적어 두고 수정을 요청하면 됩니다.
  • 법률 검색 MCP (예: law_mcp): 조문 번호·최신 조문 확인용 보조. RAG를 대체하지 않으며, 답변의 최종 검증은 사람이 합니다.
  • 브라우저·캡처 QA: 배포 전 실제로 열어 보고 스마트폰에서도 목차가 쓸 만한지 확인하세요.
